O diagrama de classe é usado para a construção de projetos com orientação a objetos como metodologia, onde são exibidas as classes que compõem o projeto no todo e como elas se relacionam entre si. Segundo o autor (Sommerville, 2011, p. 90): 
Os diagramas de classe em UML podem ser expressos em diferentes níveis de detalhamento. Quando você está desenvolvendo um modelo, o primeiro estágio geralmente é o de olhar para o mundo, identificar os objetos essenciais e representá-los como classes. A maneira mais simples de fazer isso é escrever o nome da classe em uma caixa. Você também pode simplesmente observar a existência de uma associação, traçando uma linha entre as classes.
---------------------------

O Diagrama de caso de uso deve apresentar as possíveis interações entre os atores do sistema. E tem como função facilitar a compreensão os comportamentos externos do sistema utilizando de uma linguagem simples. De todos os diagramas da UML, é o mais abstrato e com isso é o mais flexível e informal. Segundo (Sommerville, 2011, p. 74):
Atores, que podem ser pessoas ou outros sistemas, são representados como figuras ‘palito’. Cada classe de interação é representada por uma elipse. Linhas fazem a ligação entre os atores e a interação. Opcionalmente, pontas de flechas podem ser adicionadas às linhas para mostrar como a interação se inicia.
-----------------------------------


Os Diagramas de Atividades têm como função, mostrar as atividades envolvidas nos processos do projeto. Um Diagrama de Atividades é uma variação de uma máquina de estados, na qual os estados são as Atividades que representam a execução de operações e as Transições são disparadas pela conclusão destas operações. Segundo (Sommerville, 2011, p. 85):
Os diagramas de atividades são destinados a mostrar as atividades que compõem um processo de sistema e o fluxo de controle de uma atividade para a outra. O início de um processo é indicado por um círculo preenchido; o fim, por um círculo preenchido dentro de outro círculo. Os retângulos com cantos arredondados representam atividades, ou seja, os subprocessos específicos que devem ser realizados.
------------------------------------------------------------------------

Após a criação das diagramações descritas acima, também foi necessário a criação de um documento de mostrasse os relacionamentos do sistema, mas não de suas classes e objetos e sim de seu banco de dados e como suas tabelas iriam se relacionar entre elas e quais seriam seus respectivos atributos. A realização de tal documento é a criação de um modelo conceitual do banco de dados do projeto, esse modelo conceitual também pode ser chamado de DER (Diagrama Entidade-Relacionamento). Segundo a autora (Silva, 2011, p. 25):
Modelo Conceitual registra que dados podem aparecer no banco de dados, mas não registra como esses dados estão armazenados em nível de SGBD. A técnica mais difundida de modelagem conceitual é a abordagem entidade-relacionamento (ER). Nessa técnica, um modelo conceitual é usualmente representado através de um diagrama, chamado diagrama entidade-relacionamento DER.
